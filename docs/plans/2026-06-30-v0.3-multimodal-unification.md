# v0.3 架构变更计划：统一多模态模型

## 变更背景

v0.3 原架构采用三路提取：文本 LLM（DeepSeek）→ OCR+LLM（RapidOCR + DeepSeek）→ VLM 切片（Qwen-VL-Max）。三路之间通过 OCR 质量评分（0.45/0.72 阈值）切换，逻辑复杂，且 OCR 作为中间步骤损失了排版和语义信息。

小米 MiMo-V2.5 等国产多模态模型原生支持文本+图片输入，单次调用即可完成识别和语义理解，成本仅 0.7 元/百万输入 token（DeepSeek 2 元，Qwen-VL-Max 3 元）。统一到一个多模态模型可以同时降低成本和复杂度。

## 架构对比

| 维度 | 旧架构（三路） | 新架构（统一多模态） |
| --- | --- | --- |
| Provider 接口 | LLMProvider + VLMProvider + OCRProvider | MultimodalProvider |
| 提取路径 | 文本LLM / OCR+LLM / VLM切片 | 单路：有图传图，无图传文本 |
| OCR | RapidOCR（必选） | 去掉（模型直接识别图片） |
| 质量评分 | OCR 质量评分（5 维度加权） | 去掉（不再需要） |
| 成本门控 | 三层（预过滤+OCR质量+预算） | 两层（预过滤+预算） |
| 默认模型 | DeepSeek + Qwen-VL-Max | MiMo-V2.5 |
| 状态机 | candidate→ocr_done→extracted | candidate→extracted |
| run_logs | ocr_count/llm_count/vlm_count | model_count |

## 保留的模块

| 模块 | 保留原因 |
| --- | --- |
| Stitcher（短图拼接） | 微信公众号长图被拆成多张短图，需拼回原图 |
| Slicer（长图切片） | 超长图超过模型分辨率限制（通常 2000-4000px），仍需切片 |
| BudgetManager | 每日成本上限控制 |
| vlm_merge.py | 切片结果合并去重逻辑（重命名为 merge.py） |

## 开发任务分解

### Task 1: MultimodalProvider 接口 + OpenAI 兼容基类

**文件**: `providers/multimodal/base.py`, `providers/multimodal/openai_compatible.py`

**接口设计**:

```python
class MultimodalProvider(Protocol):
    name: str
    def extract_jobs(
        self,
        text: str | None,
        images: list[ImageSlice] | None,
        meta: ArticleMeta,
    ) -> ExtractionResult: ...
    def close(self) -> None: ...
```

**OpenAI 兼容基类**: 抽取 HTTP 调用、重试、JSON 解析、成本估算等公共逻辑，MiMo/Qwen/OpenAI 实现只需提供 base_url、model、价格参数。

**测试**: mock HTTP 验证请求体格式、重试逻辑、JSON 解析、成本计算。

### Task 2: MiMo-V2.5 实现

**文件**: `providers/multimodal/mimo.py`

- base_url: `https://api.xiaomimimo.com/v1`
- model: `mimo-v2.5`
- 输入价格: 0.7 元/百万 token
- 输出价格: 2.1 元/百万 token
- OpenAI 兼容协议，图片以 base64 或 URL 传入

**测试**: mock API 验证请求格式和响应解析。

### Task 3: Qwen-VL-Max 迁移

**文件**: `providers/multimodal/qwen_vl.py`

从 `providers/vlm/qwen_vl.py` 迁移，适配 MultimodalProvider 接口。保留原有的切片调用逻辑，但接口统一为 `extract_jobs(text, images, meta)`。

**测试**: 复用现有 qwen_vl 测试，适配新接口。

### Task 4: 工厂 + 配置更新

**文件**: `providers/factory.py`, `config/loader.py`, `config/.env`

- `create_multimodal_provider()` 替代 `create_llm_provider()` + `create_vlm_provider()` + `create_ocr_provider()`
- 配置项: `MULTIMODAL_PROVIDER/MULTIMODAL_API_KEY/MULTIMODAL_MODEL/MULTIMODAL_BASE_URL`
- rules.yaml: `daily_vlm_budget_cny` → `daily_model_budget_cny`

**测试**: 工厂切换测试、配置加载测试。

### Task 5: Extractor 简化为单路提取

**文件**: `modules/extractor/extractor.py`

核心变更:
- 去掉 `_extract_with_llm()`, `_extract_with_ocr()`, `_extract_with_vlm()` 三个方法
- 新增 `_extract_with_multimodal()` 统一方法
- 逻辑: stitcher 拼接 → slicer 切片（如需）→ multimodal.extract_jobs(text, images, meta)
- 去掉 OCR 质量评分和三路切换逻辑
- 预算检查: 每次调用前检查 budget_manager.remaining > 0

**测试**: mock MultimodalProvider 验证单路提取流程、预算耗尽转 need_review、切片合并去重。

### Task 6: 领域模型 + 状态机 + 数据库更新

**文件**: `domain/models.py`, `domain/status.py`, `modules/storage/schema.sql`, `modules/storage/repository.py`

- ExtractionResult: `llm_calls/vlm_calls/ocr_calls` → `model_calls/cost_estimate`
- Status: 移除 `OCR_DONE`, `ERROR_OCR`
- run_logs: `ocr_count/llm_count/vlm_count` → `model_count`
- repository: 更新 `_RUN_UPDATE_COLUMNS`, `query_pending_articles`

**测试**: 模型字段测试、状态迁移测试、数据库读写测试。

### Task 7: Runner + CLI 适配

**文件**: `pipeline/runner.py`, `main.py`

- Runner: `_init_extractor()` 简化为初始化 MultimodalProvider + Stitcher + Slicer + BudgetManager
- Runner: close() 清理 MultimodalProvider
- CLI: `review` 命令保持不变
- CLI: `run --stats` 展示 model_count/cost_estimate

**测试**: Runner 初始化测试、close 资源清理测试。

### Task 8: 清理旧文件 + 更新测试

- 删除 `providers/llm/`, `providers/vlm/`, `providers/ocr/`
- 删除 `extractor/ocr_quality.py`
- 合并 Prompt 模板
- 更新所有引用旧接口的测试
- 运行全量测试确保通过

## 迁移策略

1. 数据库迁移: run_logs 表新增 model_count 列，旧数据 ocr_count/llm_count/vlm_count 不迁移（历史数据保留原样）
2. 配置迁移: .env 文件更新为 MULTIMODAL_*，旧 LLM_*/VLM_*/OCR_* 配置可手动删除
3. 状态迁移: 已有 ocr_done 状态的文章强制迁移到 candidate（重新提取）
4. 向后兼容: factory.py 检测到旧配置项时输出警告并提示更新
